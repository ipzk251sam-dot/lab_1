# Принципи програмування в проєкті Banking System

Цей файл демонструє дотримання ключових принципів розробки програмного забезпечення у даному проєкті.

## 1. Single Responsibility Principle (SRP) — Принцип єдиної відповідальності
Кожен клас повинен мати лише одну причину для змін. Ми розділили логіку даних, бізнес-логіку та роботу з файлами.

* **Опис:** * Клас `Account` відповідає виключно за збереження стану рахунку (дані).
    * Клас `FileHandler` відповідає виключно за запис та зчитування файлів.
* **Демонстрація в коді:**
    * [Клас даних Account (src/models/Account.h, рядки 7-10)](src/models/Account.h#L7-L10)
    * [Логіка роботи з файлами (src/utils/FileHandler.cpp, рядки 6-15)](src/utils/FileHandler.cpp#L6-L15)

---

## 2. Encapsulation — Інкапсуляція
Приховування внутрішньої реалізації та захист даних від некоректного втручання.

* **Опис:** * Всі поля класу `Account` (`id`, `name`, `balance`) є `private`.
    * Доступ до них здійснюється через публічні методи (геттери).
    * Зміна балансу можлива лише через методи `deposit` та `withdraw`, які містять валідацію.
* **Демонстрація в коді:**
    * [Приватні поля (src/models/Account.h, рядок 7)](src/models/Account.h#L7)
    * [Метод withdraw з перевіркою балансу (src/models/Account.cpp, рядки 19-25)](src/models/Account.cpp#L19-L25)

## 3. Separation of Concerns (SoC) — Розділення інтересів
Програма розділена на логічні шари: Інтерфейс (UI), Сервіс (Logic) та Дані (Data).

* **Опис:** * `main.cpp` відповідає лише за взаємодію з користувачем (меню, `cin`/`cout`). Він не знає, як зберігаються дані у файлі.
    * `BankService` виступає посередником: він отримує команди від `main`, обробляє їх і звертається до `Account` або `FileHandler`.
* **Демонстрація в коді:**
    * [Інтерфейс користувача (src/main.cpp, рядки 4-12)](src/main.cpp#L4-L12)
    * [Сервісний шар (src/services/BankService.cpp, рядки 10-16)](src/services/BankService.cpp#L10-L16)

## 4. DRY (Don't Repeat Yourself) — Не повторюй себе
Уникнення дублювання коду шляхом винесення спільної логіки у функції.

* **Опис:** * Ми не пишемо код пошуку рахунку в масиві кожного разу, коли хочемо зробити депозит або зняття коштів.
    * Замість цього ми створили один метод `getAccount(id)`, який використовується повторно.
* **Демонстрація в коді:**
    * [Визначення методу пошуку (src/services/BankService.cpp, рядки 18-25)](src/services/BankService.cpp#L18-L25)
    * [Використання методу в main (src/main.cpp, рядок 37)](src/main.cpp#L37)

## 5. KISS (Keep It Simple, Stupid) — Не ускладнюй
Використання простих рішень замість складних архітектур, де це можливо.

* **Опис:** * Для збереження бази даних використовується простий текстовий формат із роздільником `|`.
    * Це дозволяє легко переглядати та редагувати дані вручну без використання складних бібліотек (як SQL або JSON парсерів) для навчальних цілей.
* **Демонстрація в коді:**
    * [Простий формат запису у файл (src/utils/FileHandler.cpp, рядок 10)](src/utils/FileHandler.cpp#L10)
